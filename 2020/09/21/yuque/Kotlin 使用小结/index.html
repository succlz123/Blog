    let、apply、with 的区别和应用场景 | let、with、apply 都是标准库函数，它们的主要区别在 lambda 参数类型定义不同。apply、with 的 lambda 参数是 T 的扩展函数，因此在 lambda 内使用 this 引用接收者对象，而 let 的 lambda 参数是参数为 T 的高阶函数，因此 lambda 内使用 it 引用唯一参数。

let block: (T) -> R - 返回一个新的对象，let 的 block 的最后

apply - block: T.() -> Unit - 返回的是 T 当前.的对象

run - block: T.() -> R - 返回一个新的对象，run 的 block 的最后 |
| reified - 具体化 关键字 | 可以拿到泛型的 k class 信息,  kclass 就是 kotlin 对 java class 的一层封装 - 核心是编译阶段生成的 metadata 注解, 里面包括 class 很多信息
使用
a. 在泛型类型前面增加 reified
b. 在方法前面增加 inline（必需的）
原理

  • Kotlin 编译器会将 reified 方法asType内联(inline)到调用的地方(call-site)
  • 方法被内联到调用的地方后，泛型 T 会被替换成具体的类型

所以 reified 使得泛型的方法假装在运行时能够获取泛型的类信息 ,   而且 基于 inline 的  reified 是会智能根据顺序 内联相应代码 |
| 委托机制 | Kotlin 委托的语法关键字是 by，其本质上是面向编译器的语法糖，三种委托（类委托、对象委托和局部变量委托）在编译时都会转化为 “无糖语法”。例如类委托：编译器会实现基础接口的所有方法，并直接委托给基础对象来处理。例如对象委托和局部变量委托：在编译时会生成辅助属性（prop$degelate），而属性 / 变量的 getter() 和 setter() 方法只是简单地委托给辅助属性的 getValue() 和 setValue() 处理。 |

各种注解

JvmField   不要生成 getter/setter 并将其作为公开字段（public）
JvmName 用注解名字替换
JvmMultifileClass 生成一个多文件类，该文件具有在此文件中声明的顶级函数和属性作为其中的一部分 就是最后合并到一个 clas 里
JvmOverloads   默认参数值的重载
Strictfp 浮点数精度控制
Synchronized Volatile Transient
JvmDefault
kt 的默认 interface 默认方法是通过生成 静态内部类 DefaultImpls 实现
这样与 java 互操作就 看着有点奇怪, 加这个注解, 需要 1.8 +  Xjvm-default=enable 开启
因为 java8 通过在 方法前加 default 可以支持 接口方法, 最后 kt 字节码 就是 java8 样子了
还可以用  1.8 + -Xjvm-default=compatibility 开启兼容模式, 这样没加就是默认, 加了就是新的, 然后 没加的   静态方法就变成 调用 他自己的  default 方法
