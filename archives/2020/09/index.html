原理

  • Kotlin 编译器会将 reified 方法asType内联(inline)到调用的地方(call-site)
  • 方法被内联到调用的地方后，泛型 T 会被替换成具体的类型

所以 reified 使得泛型的方法假装在运行时能够获取泛型的类信息 ,   而且 基于 inline 的  reified 是会智能根据顺序 内联相应代码 |

JvmDefault
kt 的默认 interface 默认方法是通过生成 静态内部类 DefaultImpls 实现
这样与 java 互操作就 看着有点奇怪, 加这个注解, 需要 1.8 +  Xjvm-default=enable 开启
因为 java8 通过在 方法前加 default 可以支持 接口方法, 最后 kt 字节码 就是 java8 样子了
还可以用  1.8 + -Xjvm-default=compatibility 开启兼容模式, 这样没加就是默认, 加了就是新的, 然后 没加的   静态方法就变成 调用 他自己的  default 方法

什么是启动框架

把 App 启动时候 Application 那一堆东西按照 Task 分类，根据依赖关系和优先级排序  (又比如 Gradle 中 Task 的前后依赖也差不多)

目的

  1. 很多 Task 其实可以延后执行的，只执行必要的任务。
  2. 按照 Task 组织后也方便进行测量耗时，进而优化。

    收益

加快启动速度，对于 Android 应用来说这种框架原理很简单，但是却是必须的。
